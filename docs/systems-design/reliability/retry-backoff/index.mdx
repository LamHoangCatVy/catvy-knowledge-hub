# Retry and Backoff Strategies in Distributed Systems

When a service call fails, the natural reaction is to retry. But uncontrolled retries are one of the most common causes of cascading failure in distributed systems. If 1000 concurrent requests hit an overloaded service, and each retries 3 times, the service receives 4000 requests instead of 1000 — worsening the original overload. This is called a retry storm.

## When to Retry

Not every failure should be retried. The fundamental principle: only retry when there is a reasonable basis to believe the next attempt will succeed. This means retries are appropriate for transient failures — network errors, timeouts, rate limiting — but not for permanent failures — invalid data, authentication failures, non-existent resources.

Classifying failures is the first step: transient or permanent. HTTP status codes provide guidance: 429 (Too Many Requests) and 503 (Service Unavailable) are transient and should be retried. 400 (Bad Request) and 401 (Unauthorized) are permanent and should not be retried. 500 (Internal Server Error) is a gray area — it may be transient (temporary glitch) or permanent (logic error). For 500s, a limited number of retries is reasonable.

## Exponential Backoff with Jitter

When retrying, the wait time between attempts should increase exponentially. This gives the target service time to recover. The basic formula:

```
wait_time = base_delay * 2^attempt
```

With a base_delay of 100ms, the first retry waits 100ms, the second 200ms, the third 400ms, the fourth 800ms, and so on. However, pure exponential backoff has a problem: if many clients fail simultaneously, they will all retry at the same time, creating synchronized waves of requests. This is called the thundering herd effect.

Jitter solves this problem by adding random noise to the wait time:

```
wait_time = random_between(0, base_delay * 2^attempt)
```

With jitter, clients retry at different times, spreading the load more evenly over time. Full jitter — picking randomly within the entire interval — is more effective than equal jitter — halving the interval and adding small noise — at mitigating thundering herd.

## Retry Budget

Instead of limiting the number of retries per individual request, a retry budget limits the retry rate across the entire system. If the retry rate exceeds a certain threshold — for example, 10 percent of total requests — the system stops retrying entirely, even for requests that have not yet reached their individual retry limit.

A retry budget acts as a system-wide fuse. When a service begins failing broadly, the retry budget prevents a retry storm by sacrificing some individual requests to protect the entire system. This is an application of the principle of controlled sacrifice: it is better to reject some requests immediately than to let the entire system collapse under retry pressure.

## Idempotency

Every retry mechanism creates the possibility of a request being processed multiple times. If a request is sent, the server processes it successfully but the response is lost on the way back, the client will retry — and the server will process the same request a second time. If the operation is not idempotent, this leads to inconsistent state: charges deducted twice, emails sent twice, records created twice.

An idempotency key is the defense mechanism. Each request carries a unique key generated by the client. The server stores this key along with the result of the request. If the same key is sent again, the server returns the stored result instead of reprocessing. The idempotency key should be stored for a duration long enough to cover the retry window — typically 24 hours.

## Design Principles

Retry strategies rest on three principles. First, retry selectively — not every failure should be retried, and retrying permanent failures wastes resources and slows error feedback. Second, backoff with jitter is mandatory — exponential backoff without jitter creates thundering herd, worsening the incident. Third, system-wide retry budget — global retry limits prevent retry storms from cascading, even when individual limits would still allow retries.
