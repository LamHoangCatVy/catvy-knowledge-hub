Every RAG system needs a vector database. The question is: which one? After deploying all three major options in production — Pinecone, Weaviate, and pgvector — I have opinions. Here's the comparison I wish I'd had before making the choice.

{/* truncate */}

## The Three Contenders

### Pinecone
The managed, serverless vector database. You create an index, upload embeddings, and query. No infrastructure to manage. Scales to billions of vectors. Pay per pod or serverless by usage. The most "just works" option — and the most expensive at scale.

### Weaviate
The open-source, feature-rich option. Self-hosted or managed cloud. Supports vector search, hybrid search (BM25 + vectors), and generative search (retrieve then generate). GraphQL-native API. The most feature-complete option — and the most complex to operate.

### pgvector
The PostgreSQL extension that adds vector search to your existing database. No new infrastructure. Vectors live alongside your relational data. Simple, familiar, and zero additional operational overhead. The most pragmatic option — and the least optimized for pure vector search at extreme scale.

## Performance at Different Scales

I benchmarked all three with the same dataset: 1 million document chunks embedded with `text-embedding-3-large` (3072 dimensions).

### 10K Vectors (Prototype Scale)

| Metric | Pinecone | Weaviate | pgvector |
|---|---|---|---|
| Query latency (p95) | 45ms | 60ms | 35ms |
| Index build time | 22s | 35s | 18s |
| Recall@10 | 98.2% | 97.8% | 96.1% |

At prototype scale, all three are fast enough. pgvector is slightly faster because vectors live in the same process as your application. Pinecone has the best recall because its indexing is purpose-built.

### 100K Vectors (Production Scale)

| Metric | Pinecone | Weaviate | pgvector |
|---|---|---|---|
| Query latency (p95) | 85ms | 110ms | 140ms |
| Recall@10 | 97.8% | 97.5% | 92.3% |
| Memory usage | 2.1 GB (managed) | 4.8 GB | 3.2 GB |

At 100K vectors, pgvector's default IVFFlat index starts showing its limitations — recall drops noticeably. Weaviate's HNSW index holds strong. Pinecone's proprietary index maintains the best balance.

### 10M Vectors (Enterprise Scale)

| Metric | Pinecone | Weaviate | pgvector |
|---|---|---|---|
| Query latency (p95) | 120ms | 195ms | 380ms |
| Recall@10 | 97.1% | 96.8% | 89.7% |
| Monthly cost | $850 | $420 (self-hosted) | $180 (RDS) |

At 10M vectors, pgvector with IVFFlat struggles unless you switch to HNSW indexing (added in pgvector 0.6). With HNSW, pgvector recall improves to 94% but query latency stays around 250ms. Pinecone remains the fastest, but the cost gap widens significantly.

## The Hybrid Search Gap

This is where the comparison gets interesting. For production RAG, vector-only search is insufficient — hybrid search (keyword + semantic) consistently improves retrieval accuracy by 15-30%.

- **Pinecone**: No native BM25 support. You need an external keyword index (Elasticsearch, Typesense) and merge results yourself.
- **Weaviate**: Native hybrid search built in. BM25 and vector scores combined via reciprocal rank fusion. Zero extra infrastructure.
- **pgvector**: You already have PostgreSQL. BM25 via `tsvector` and `tsquery` — PostgreSQL's built-in full-text search. Hybrid search in a single SQL query.

For hybrid search, pgvector + PostgreSQL full-text search is shockingly good. The combination of `ts_rank` (keyword relevance) and `<=>` (cosine distance) in a single query with configurable weighting gives you hybrid search without any external infrastructure.

## Cost at Scale

### Pinecone (Serverless)
- Storage: $0.33/GB/month for vectors
- Writes: $0.25/1M write units
- Reads: $0.50/1M read units
- For 10M vectors (3072-dim): ~$350-850/month depending on query volume

### Weaviate (Self-Hosted on AWS)
- EC2 (c6g.2xlarge): $200/month reserved
- EBS (500GB gp3): $40/month
- Total: ~$240/month + operational overhead

### pgvector (RDS)
- RDS (db.r6g.large): $150/month reserved
- EBS (500GB gp3): $40/month
- Total: ~$190/month — and you're using the same database for relational data

## The Decision Framework

After deploying all three, here's how I choose:

### Choose Pinecone When
- You need to scale to hundreds of millions of vectors.
- Your team has zero appetite for infrastructure management.
- Budget is flexible and speed-to-market matters more than cost optimization.
- You don't need hybrid search, or you're willing to run a separate keyword index.

### Choose Weaviate When
- You need native hybrid search.
- You want a rich feature set (generative search, multi-tenancy, classification).
- You're comfortable managing infrastructure (or paying for Weaviate Cloud).
- You need multimodal search (text + image vectors in the same index).

### Choose pgvector When
- You have fewer than 10 million vectors.
- You already use PostgreSQL and want to minimize new infrastructure.
- Hybrid search is critical and you want it in a single database.
- Operational simplicity is a top priority.
- You're budget-conscious and have PostgreSQL expertise on the team.

## What I Actually Use

At my company, we run pgvector for our main RAG pipeline (~2M documents, hybrid search, PostgreSQL full-text + pgvector HNSW). It's fast enough, cheap enough, and our team already knows PostgreSQL.

For a separate project with 50M+ documents and strict latency requirements, we use Pinecone. The managed service premium is worth it when the alternative is building and maintaining a distributed vector search infrastructure.

Weaviate is my recommendation for teams starting fresh who want maximum features without managing separate systems. But if you already run PostgreSQL, try pgvector before reaching for a dedicated vector database. You might be surprised how far it takes you.

## The Trend I'm Watching

PostgreSQL extensions are eating the specialized database market. pgvector, pgai (AI functions), pg_partman (partitioning), and pg_cron (scheduling) mean you can run your entire RAG stack — relational data, vector search, full-text search, and background jobs — on a single PostgreSQL instance. The operational simplicity of one database that does everything is compelling, even if each individual feature isn't best-in-class.

For 90% of RAG use cases, that's the right trade-off.
